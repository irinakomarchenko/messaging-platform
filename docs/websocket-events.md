# WebSocket Events

## Overview

This document describes WebSocket events for real-time messaging.

REST API is used for registration, login, search, channel management and history reading.
WebSocket is used for real-time message exchange, delivery acknowledgements, heartbeat and presence events.

## WebSocket Endpoint

```text
/ws/messages
```

In the first version the client connects directly to `messaging-service`.

Later the connection can go through API Gateway or Load Balancer.

---

# Why WebSocket

Polling is not suitable for high-scale real-time messaging because clients repeatedly ask the server for updates, often receiving empty responses.

WebSocket provides a persistent bidirectional connection:

```text
Client <------ WebSocket connection ------> Messaging Service
```

This allows the server to push messages to online users immediately.

---

# General Event Structure

All WebSocket messages use JSON.

## Client-to-Server Event Format

```json
{
  "type": "EVENT_TYPE",
  "payload": {}
}
```

## Server-to-Client Event Format

```json
{
  "type": "EVENT_TYPE",
  "payload": {}
}
```

---

# Event Types

## Client-to-Server Events

| Event Type     | Purpose                                         |
| -------------- | ----------------------------------------------- |
| `SEND_MESSAGE` | Client sends a message                          |
| `HEARTBEAT`    | Client confirms that connection is alive        |
| `SYNC_REQUEST` | Client asks for missed messages after reconnect |

## Server-to-Client Events

| Event Type         | Purpose                                          |
| ------------------ | ------------------------------------------------ |
| `MESSAGE_ACK`      | Server confirms that message was accepted/stored |
| `MESSAGE_RECEIVED` | Server delivers a message to recipient           |
| `PRESENCE_CHANGED` | Server notifies about online/offline status      |
| `SYNC_RESPONSE`    | Server returns missed messages                   |
| `ERROR`            | Server returns WebSocket-level error             |

---

# SEND_MESSAGE

Client sends a message through WebSocket.

## Client Event

```json
{
  "type": "SEND_MESSAGE",
  "payload": {
    "clientMessageId": "client-msg-123",
    "conversationId": "private-1-2",
    "conversationType": "PRIVATE_CHAT",
    "senderId": 1,
    "text": "Hello"
  }
}
```

## Fields

| Field              | Description                               |
| ------------------ | ----------------------------------------- |
| `clientMessageId`  | Unique id generated by client             |
| `conversationId`   | Target conversation id                    |
| `conversationType` | `PRIVATE_CHAT`, `GROUP_CHAT` or `CHANNEL` |
| `senderId`         | Id of sender                              |
| `text`             | Message text                              |

## Validation Rules

| Field              | Rule                            |
| ------------------ | ------------------------------- |
| `clientMessageId`  | required                        |
| `conversationId`   | required                        |
| `conversationType` | required                        |
| `senderId`         | required                        |
| `text`             | required, max 10,000 characters |

## Implementation Flow

```text
1. Client sends SEND_MESSAGE.
2. messaging-service validates event.
3. messaging-service stores message through chat-history-service.
4. chat-history-service returns serverMessageId.
5. messaging-service sends MESSAGE_ACK to sender.
6. messaging-service resolves recipients.
7. messaging-service checks online recipients through connection-management-service.
8. messaging-service delivers MESSAGE_RECEIVED to online recipients.
9. Offline users receive messages later through sync.
```

# MESSAGE_ACK

Server confirms that message was accepted and stored.

## Server Event

```json
{
  "type": "MESSAGE_ACK",
  "payload": {
    "clientMessageId": "client-msg-123",
    "serverMessageId": 1000,
    "conversationId": "private-1-2",
    "status": "STORED",
    "createdAt": "2026-06-08T12:00:00Z"
  }
}
```

## ACK Statuses

| Status      | Meaning                                       |
| ----------- | --------------------------------------------- |
| `RECEIVED`  | messaging-service received event              |
| `STORED`    | message was stored in chat-history-service    |
| `DELIVERED` | message was delivered to recipient connection |
| `FAILED`    | message processing failed                     |

For the first version, the main status is `STORED`.

Later we can add full delivery status:

```text
SENT -> STORED -> DELIVERED -> READ
```

`READ` is out of scope for the first implementation.

---

# MESSAGE_RECEIVED

Server delivers a message to an online recipient.

## Server Event

```json
{
  "type": "MESSAGE_RECEIVED",
  "payload": {
    "serverMessageId": 1000,
    "conversationId": "private-1-2",
    "conversationType": "PRIVATE_CHAT",
    "senderId": 1,
    "text": "Hello",
    "createdAt": "2026-06-08T12:00:00Z"
  }
}
```

## Delivery Rule

If the recipient is online, `messaging-service` sends this event through the active WebSocket connection.

If the recipient is offline, the message remains stored in `chat-history-service` and is returned later through sync.

# HEARTBEAT

Client sends heartbeat to keep connection alive.

## Client Event

```json
{
  "type": "HEARTBEAT",
  "payload": {
    "userId": 1,
    "connectionId": "ws-abc-123",
    "sentAt": "2026-06-08T12:00:00Z"
  }
}
```

## Server Event

```json
{
  "type": "HEARTBEAT_ACK",
  "payload": {
    "userId": 1,
    "connectionId": "ws-abc-123",
    "status": "ALIVE",
    "receivedAt": "2026-06-08T12:00:01Z"
  }
}
```

## Implementation Notes

The `messaging-service` receives heartbeat and updates connection state in `connection-management-service`.

In the first version heartbeat can be simple.

Later production-like behavior:

* heartbeat every 30 seconds;
* connection TTL in Redis;
* mark user offline when heartbeat expires;
* remove stale connections automatically.

# PRESENCE_CHANGED

Server notifies clients about online/offline status changes.

## Server Event

```json
{
  "type": "PRESENCE_CHANGED",
  "payload": {
    "userId": 2,
    "status": "ONLINE",
    "changedAt": "2026-06-08T12:00:00Z"
  }
}
```

## Presence Statuses

```text
ONLINE
OFFLINE
```

## First Version

Presence events can be implemented later after basic message delivery works.

The first implementation only needs to store and return user presence through REST API.

# SYNC_REQUEST

Client asks for missed messages after reconnect.

## Client Event

```json
{
  "type": "SYNC_REQUEST",
  "payload": {
    "userId": 1,
    "conversationId": "private-1-2",
    "afterMessageId": 1000
  }
}
```

## Server Event

```json
{
  "type": "SYNC_RESPONSE",
  "payload": {
    "conversationId": "private-1-2",
    "afterMessageId": 1000,
    "messages": [
      {
        "serverMessageId": 1001,
        "senderId": 2,
        "text": "Hi",
        "createdAt": "2026-06-08T12:01:00Z"
      }
    ]
  }
}
```

## MVP Decision

For the first implementation, missed messages sync can be implemented as REST:

```text
GET /api/v1/conversations/{conversationId}/messages/sync?afterMessageId=1000
```

WebSocket `SYNC_REQUEST` can be added later.

# ERROR

Server returns error for invalid WebSocket events.

## Server Event

```json
{
  "type": "ERROR",
  "payload": {
    "code": "INVALID_MESSAGE",
    "message": "Message text must not be blank",
    "clientMessageId": "client-msg-123"
  }
}
```

## Common Error Codes

| Code                 | Meaning                            |
| -------------------- | ---------------------------------- |
| `INVALID_EVENT_TYPE` | Unknown event type                 |
| `INVALID_MESSAGE`    | Invalid payload                    |
| `UNAUTHORIZED`       | Invalid or missing authentication  |
| `FORBIDDEN`          | User has no access to conversation |
| `MESSAGE_TOO_LARGE`  | Message exceeds max size           |
| `INTERNAL_ERROR`     | Unexpected server error            |

# Idempotency

The client must generate `clientMessageId` for every message.

The pair below should be unique:

```text
senderId + clientMessageId
```

This prevents duplicate messages when the client retries after network problems.

Example:

```text
User sends message.
Network fails before ACK is received.
Client retries the same SEND_MESSAGE with the same clientMessageId.
Server detects duplicate and returns the original serverMessageId.
```

---

# Ordering

The system guarantees ordering inside one conversation, not globally across all conversations.

Recommended production strategy:

```text
partition key = conversationId
ordering key = serverMessageId or createdAt
```

For the first H2 version:

```text
ORDER BY created_at ASC, id ASC
```

---

# Store First, Deliver After

The correct processing order is:

```text
1. Store message.
2. Send ACK to sender.
3. Deliver to online recipients.
4. Offline users sync later.
```

This prevents a situation where a recipient sees a message that was not persisted in history.

---

# Future Broker-Based Delivery

In the first version, `messaging-service` can deliver messages directly.

Later, when there are many messaging-service instances:

```text
messaging-service-1 receives message
chat-history-service stores message
messaging-service-1 publishes delivery event to broker
messaging-service-2 receives event
messaging-service-2 sends message to connected recipient
```

Possible broker choices:

* Kafka;
* NATS;
* RabbitMQ;
* Redis Pub/Sub for simpler cases.
